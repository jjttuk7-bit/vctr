## 2026-05-16 | 서비스명 KNow 확정

- **결정**: 서비스명을 **KNow**로 확정. 도메인은 `k-now.co` 또는 `knowkorea.com` 우선 검토.
- **이유**: K(한국) + Now(지금) + Know(알다) 세 가지 의미가 동시에 담김. 짧고 발음이 쉬워 글로벌 독자에게 유리. 슬로건 "KNow Korea, before anyone else"가 자연스럽게 연결됨. 해시태그 #KNow가 간결하고 기억하기 쉬움.
- **대안**: KWave Daily(너무 설명적), Seoul Signal(서울 한정 느낌), HanPulse(Han 의미 불명확)
- **번복 조건**: 도메인 확보 불가 시 HanPulse 또는 Seoul Signal로 대체 검토.

## 2026-05-16 | 번역 ❌ 팩트 기반 재작성 ✅

- **결정**: 한국어 원문을 영어로 번역하지 않고, 팩트(누가·언제·무엇을·수치)만 추출해서 LLM이 완전히 새로운 영문 기사를 작성하는 방식 채택
- **이유**: 번역은 2차 저작물로 원저작권자 허락 없이 상업적 게시 불가. 팩트에는 저작권이 없으므로 팩트만 활용하면 독립적 창작물이 됨. AP·Reuters·BBC가 같은 뉴스를 각자 다르게 쓰는 것과 동일한 원리.
- **대안**: 원문 번역 후 게시 (저작권 위반 리스크 큼)
- **번복 조건**: 없음. 팩트 재작성은 저작권 안전의 핵심 원칙.

## 2026-05-16 | AI 이미지 생성 ❌ Unsplash + OG 자동생성 ✅

- **결정**: DALL-E 3 등 AI 이미지 생성을 사용하지 않고, 기사 본문은 Unsplash API(분위기 이미지), SNS/OG는 @vercel/og 자동생성으로 처리
- **이유**: AI 이미지는 월 ~$60 발생. Unsplash는 무료(월 50,000 req), @vercel/og는 완전 무료. 분위기 이미지가 특정 인물 사진보다 저작권·초상권 리스크가 없고, 브랜드 일관성 측면에서 OG 자동생성이 오히려 더 감각적.
- **대안**: DALL-E 3 (월 $60), 기사 이미지 무단 사용 (저작권 위반)
- **번복 조건**: 월 방문자 100만 돌파 후 이미지 품질이 수익에 직결된다는 데이터가 있을 때 AI 이미지 도입 검토.

## 2026-05-16 | 카테고리별 프롬프트 분리 (9개 변형)

- **대안**: 단일 프롬프트로 카테고리 명시 (품질 일관성 저하)
- **번복 조건**: 9개 관리 복잡도가 너무 높을 경우 유사 카테고리 묶기 (Beauty+Fashion, Pop+Entertainment 등)

## 2026-05-16 | SQLite DB Git 커밋 허용 (MVP)

- **결정**: `data/kwave.db` 파일을 .gitignore에서 제외하고 Git 추적 대상에 포함.
- **이유**: GitHub Actions는 매 실행 클린 환경에서 시작. DB를 커밋하지 않으면 누적 기사 데이터가 매일 초기화됨. 기사 중복 방지(URL 해시 기반)도 DB가 유지돼야 작동.
- **대안 A**: GitHub Actions Cache (캐시 만료 시 데이터 소실 위험)
- **대안 B**: PostgreSQL/Supabase (외부 서비스 의존, MVP 단계 과잉)
- **번복 조건**: DB 파일이 50MB 초과하거나 Git push 속도 현저히 저하 시 Actions Cache로 전환.

## 2026-05-16 | Unsplash 이미지 키워드 LLM 자동 생성

- **결정**: Unsplash 검색 키워드를 미리 정의한 기본값 + LLM이 기사 내용 보고 생성하는 `unsplash_keywords` 필드를 함께 활용.
- **이유**: 미리 정의된 키워드는 일관성은 있지만 기사 내용과 맞지 않을 수 있음. LLM이 기사 분위기에 맞는 키워드를 생성하면 더 적절한 이미지 매칭 가능.
- **핵심 규칙**: unsplash_keywords에 인물명 포함 금지 (초상권·저작권 문제). 분위기·장소·사물 키워드만.
- **대안**: 카테고리별 고정 키워드만 사용 (단순하지만 반복적)
- **번복 조건**: LLM이 인물명을 포함시키는 경우가 반복되면 코드 레벨 필터링 강화.

## 2026-05-16 | 파이프라인 구조: Collector → Processor → Publisher 3단계

- **결정**: 기존 설계(4단계: Fetcher → Analyzer → Translator → Publisher)가 아니라 3단계로 단순화. Fact Extractor는 Collector 내부 함수로 처리.
- **이유**: papermint에서 Analyzer + Translator를 Processor로 통합해서 LLM 비용 절반, 속도 2배 효과 검증. KNow도 동일한 원칙 적용. Fact Extractor는 LLM 호출이 없는 텍스트 처리 로직이므로 별도 모듈 불필요.
- **대안**: 4단계 분리 (각 단계 독립적이나 LLM 호출 2회)
- **번복 조건**: 재작성 품질 문제가 카테고리별로 현저히 다를 경우 Processor 내부 분리 검토.
